Atlas

Routing and workload allocation engine for a technical team · Crown Lift Trucks · built with Claude

Problem

A forklift-maintenance field team (Catalonia and the Balearic Islands) has to split its workload every day, month and year: which machine each technician takes, which route, which priority. It's a real Technician Routing and Scheduling Problem (TRSP in the literature), not a classroom exercise: every bad decision is one trip too many or one customer waiting too long.

Approach

The engine allocates in three layers that never overlap: a hard one that filters before scoring (usual technician, zone, certification), a real one that weighs proximity against workload fairness, and a tie-breaker that never pulls a machine out of its zone. It's solved by successive approximation and local search: moving and swapping pieces until there's no clear improvement left to make.

Build

I directed and built it with Claude without writing the production code myself: I don't come from code, I come from the problem. It's calibrated with the team's real productivity and travel data, not estimates, and cross-checked against the field's professional literature before any decision is trusted. Every change goes through a regression suite first: if it worsens the baseline, it isn't accepted.

Outcome

Today it's a working prototype the team uses informally, not an official company deployment. It's still alive: every improvement is tested against the baseline before it goes in.

Technical spec

The same move repeats in everything I build: find the point where manual judgment no longer scales, and build with AI the piece that replaces it with something measurable and repeatable. I don't write production code for the pleasure of programming: I do it because it's the fastest way to test whether a process idea works before asking someone to implement it for real.
